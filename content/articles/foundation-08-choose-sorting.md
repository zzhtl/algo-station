# 选择排序策略：根据约束做取舍

掌握排序不是默写五份代码，而是看到约束后能解释为什么选某一种。可以从数据规模、输入形态、稳定性、额外空间和最坏复杂度五个维度判断。

## 常见选择

| 场景 | 候选方案 | 原因 |
| --- | --- | --- |
| 小规模或近乎有序 | 插入排序 | 实现简单，近乎有序时接近 $O(n)$ |
| 要求稳定且接受辅助空间 | 归并排序 | 稳定，最坏 $O(n\log n)$ |
| 普通数组、重视平均性能 | 随机快速排序 | 缓存友好，平均 $O(n\log n)$ |
| 只要前 K 个或流式极值 | 堆 | 无需完整排序，维护 $O(k)$ 空间 |
| 整数值域很小 | 计数排序 | $O(n+k)$，跳过比较 |
| 链表排序 | 归并排序 | 切分和合并都可通过指针完成 |

## 不必总是完整排序

只求第 K 大可以用固定容量堆、快速选择或值域计数；只需要判断区间是否重叠，可能只按一个端点排序；数据不断到达时，维护结构比每次重排全部数据更合适。

先问“我真的需要全序吗”，常常能把 $O(n\log n)$ 降为 $O(n)$ 或 $O(n\log k)$。

## 标准库与手写

工程代码优先使用标准库排序：实现经过优化，边界更可靠。面试或训练中手写排序的价值，是练习不变量、分治、递归边界和复杂度推导，而不是在业务代码里替换标准库。

使用标准库时仍要确认：

- API 是稳定排序还是不稳定排序；
- 比较器是否完整且满足传递性；
- 排序是否会修改原输入；
- 键提取是否昂贵，能否预计算；
- 浮点数中的 NaN 如何处理。

## 一张决策清单

1. `n` 最大是多少？
2. 是否只需要 Top K、分位数或局部顺序？
3. 是否要求稳定？
4. 能使用多少额外空间？
5. 是否必须保证最坏 $O(n\log n)$？
6. 值域、位数或分布有没有额外结构？

最终目标不是说出“快速排序通常最快”，而是能把选择与题目约束逐项对应，并知道选择失效时的替代方案。
