# 排序稳定性与比较器：把顺序规则写完整

排序不只是在数字间比较大小。记录可能有多个字段，题目也可能要求按拼接结果、区间端点或自定义优先级排序。此时需要同时考虑稳定性和比较器是否合法。

## 什么是稳定排序

如果两个元素的排序键相等，排序后仍保持它们在输入中的相对次序，这个排序就是稳定的。例如先按姓名排序，再用稳定排序按班级排序，同班学生仍保留姓名顺序，因此得到“班级为主、姓名为次”的结果。

插入排序和按规则实现的归并排序可以稳定；常见原地快速排序和堆排序通常不稳定。标准库是否稳定要查对应 API，不能只凭算法名猜测。

## 多关键字比较

比较器应按主关键字、次关键字依次判断：

```text
先比较结束时间；
结束时间相同，再比较开始时间；
仍相同则视为等价。
```

不要用整数相减实现比较，极值可能溢出。Rust 可用元组的 `cmp`，Go 的 `sort.Slice` 应逐字段写 `<` 判断。

## 严格弱序

一个合法比较器至少需要保持：

- 不会出现 `a < a`；
- 若 `a < b`，就不能同时 `b < a`；
- 若 `a < b` 且 `b < c`，必须有 `a < c`。

违反传递性会让标准库排序得到不可预测结果。尤其不要让比较规则依赖会在排序过程中变化的外部状态。

## 拼接类比较

“组成最大数”可以比较字符串 `a+b` 与 `b+a`，而不是把字符串直接转成整数。排序完成后还要处理全零输入，避免输出多个前导零。

## 复盘问题

- 相同主键的元素是否要求保留原次序？
- 是否可以把多个键组成元组比较？
- 自定义比较是否满足传递性？
- 极值、空字符串、重复元素会不会破坏规则？

稳定性是一种可利用的语义，不只是排序算法表格里的一列属性；比较器则是排序正确性的组成部分，而不是语法细节。
