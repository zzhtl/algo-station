# 两端指针缩小候选：数组与字符串训练题解

两端指针最容易被写成“看起来很像模板”的代码：`left++`、`right--` 都会写，但为什么可以移动这一侧却说不清。真正要训练的是一个判断：**当前两端已经形成了一个候选答案，下一步移动哪一端，能够一次排除哪些不可能更优的组合**。

如果这个排除理由说不出来，就不要急着写双指针。很多题只是长得像两端指针，实际需要哈希、前缀和或动态规划。

## 适用场景

适合相向双指针的题，通常有一个“区间越缩越小”的候选空间，并且移动某一端有明确的排除依据。

- 有序数组中找和、找最接近的和：和太小就移动左端，和太大就移动右端。
- 容器盛水：面积受较短板限制，移动较高板不会让当前较短板参与的组合变好。
- 回文判断：两端字符必须配对，匹配后才能向内收缩。
- 排序后的去重枚举：固定一维后，用两端指针在剩余区间里找目标。

不适合的信号也要记住：如果移动左端或右端都没有单调理由，只是“试试看”，那通常还是暴力枚举的变形。

## 图解思路

```mermaid
flowchart LR
  A[候选区间 nums[left..right]] --> B[计算当前两端形成的状态]
  B --> C{能否更新答案?}
  C --> D[记录答案]
  C --> E[分析瓶颈或大小关系]
  D --> E
  E -->|左端不可能更优| F[left += 1]
  E -->|右端不可能更优| G[right -= 1]
  F --> A
  G --> A
```

写代码前先把这三句话补完整：

- `left/right` 表示还没有被排除的候选边界。
- 当前答案由 `nums[left]` 和 `nums[right]` 共同决定，或者由它们夹住的区间决定。
- 本轮移动某一端，是因为这一端继续留着也不可能得到更好的结果。

第三句话是不变量的核心。比如 #11 盛最多水的容器中，若 `height[left] <= height[right]`，当前宽度已经是 `left` 能拿到的最大宽度，而高度又被 `height[left]` 限制；继续固定 `left`、移动 `right` 只会让宽度变小，所以可以放心丢掉 `left`。

## 手写步骤

1. 初始化 `left = 0, right = n - 1`，并准备答案变量。
2. 在 `left < right` 时计算当前两端形成的候选值。
3. 根据题目目标更新答案，或判断是否已经命中目标。
4. 写出移动理由：和太小移动 `left`，和太大移动 `right`；短板在左移动 `left`，短板在右移动 `right`。
5. 如果题目要求去重，移动后再跳过连续重复值，不要在更新答案前跳。

## Go 参考骨架

```go
func maxArea(height []int) int {
	left, right := 0, len(height)-1
	ans := 0
	for left < right {
		width := right - left
		h := height[left]
		if height[right] < h {
			h = height[right]
		}
		if area := width * h; area > ans {
			ans = area
		}

		if height[left] <= height[right] {
			left++
		} else {
			right--
		}
	}
	return ans
}
```

## Rust 参考骨架

```rust
pub fn two_sum_sorted(numbers: Vec<i32>, target: i32) -> Vec<i32> {
    let (mut left, mut right) = (0usize, numbers.len() - 1);
    while left < right {
        let sum = numbers[left] + numbers[right];
        if sum == target {
            return vec![left as i32 + 1, right as i32 + 1];
        }
        if sum < target {
            left += 1;
        } else {
            right -= 1;
        }
    }
    vec![]
}
```

## 为什么这样写

暴力枚举会检查所有 `(i, j)`，时间是 $O(n^2)$。两端指针能降到 $O(n)$，靠的不是少写了一层循环，而是每一轮都能删掉一整批候选。

以有序数组两数之和为例：

- `numbers[left] + numbers[right] < target` 时，固定 `left` 再换任何更小的 `right`，和只会更小，所以 `left` 可以整体排除。
- `numbers[left] + numbers[right] > target` 时，固定 `right` 再换任何更大的 `left`，和只会更大，所以 `right` 可以整体排除。

这个“整体排除”才是相向双指针的价值。写题解或复盘时，重点也应该落在这里。

## 复杂度

- 时间复杂度：单次相向扫描是 $O(n)$；如果需要先排序，整体是 $O(n \log n)$。
- 空间复杂度：只用两个指针和少量变量时是 $O(1)$；排序是否额外占空间取决于语言和排序实现。

## 易错点

- 没有证明移动理由，只凭感觉移动，容易漏解。
- #15 三数之和中，找到答案后忘记同时跳过左右两侧重复值。
- #11 容器盛水中移动了较高的一侧，导致短板仍然限制高度，只是白白缩小宽度。
- Rust 中 `right -= 1` 前要保证 `left < right`，避免 `usize` 下溢。
