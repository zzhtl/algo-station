# 同向指针原地覆盖：数组与字符串训练题解

同向双指针解决的是一类原地整理问题：数组还要继续从左到右读，但前面一段已经可以确定最终形态。于是我们用 `read` 负责扫描原数据，用 `write` 指向下一个可写位置。

这类题的关键不是“两个指针同向移动”，而是一直维护一个清楚的不变量：**`nums[0..write)` 已经整理好，并且不会再被后续读取依赖**。

## 适用场景

看到“原地”“删除”“去重”“移动到末尾”“压缩字符串”时，先想能不能把元素分成两类：要保留的、要丢弃或延后处理的。

- 删除指定值：读到不等于目标值的元素，就写到前缀。
- 有序数组去重：读到一个“应该保留”的新值，才写入。
- 移动零：先把非零元素稳定写到前面，再把剩余位置补零。
- 字符串压缩：读指针分组统计，写指针输出字符和次数。

如果题目要求保持相对顺序，写指针只能从左往右稳定覆盖；如果不要求顺序，才可以考虑首尾交换。

## 图解思路

```mermaid
flowchart LR
  A[read 扫描当前元素] --> B{当前元素是否保留?}
  B -->|保留| C[写入 nums[write]]
  C --> D[write += 1]
  B -->|不保留| E[只移动 read]
  D --> F[read += 1]
  E --> F
  F --> A
```

写之前先明确三段区间：

- `nums[0..write)`：已经整理好的结果前缀。
- `nums[write..read)`：已经看过但不需要保留，或暂时无意义的区域。
- `nums[read..n)`：还没读取，不能提前覆盖。

只要每次写入的位置都满足 `write <= read`，就不会覆盖未来还没读到的数据。这是原地覆盖能够成立的底层原因。

## 手写步骤

1. 初始化 `write = 0`。
2. 用 `read` 从左到右扫描每个元素。
3. 判断当前元素是否应该出现在最终前缀中。
4. 如果应该保留，执行 `nums[write] = nums[read]`，再移动 `write`。
5. 扫描结束后，`write` 通常就是新长度；若题目要求补齐后缀，再处理 `write..n`。

## Go 参考骨架

```go
func removeElement(nums []int, val int) int {
	write := 0
	for read := 0; read < len(nums); read++ {
		if nums[read] != val {
			nums[write] = nums[read]
			write++
		}
	}
	return write
}
```

## Rust 参考骨架

```rust
pub fn move_zeroes(nums: &mut Vec<i32>) {
    let mut write = 0usize;
    for read in 0..nums.len() {
        if nums[read] != 0 {
            nums[write] = nums[read];
            write += 1;
        }
    }
    for i in write..nums.len() {
        nums[i] = 0;
    }
}
```

## 为什么这样写

原地删除并不是真的把元素从数组里“挖掉”，而是重新定义前缀长度。只要前 `write` 个位置满足题意，后面的值是什么通常不重要。

以 #27 删除元素为例，读指针每遇到一个不等于 `val` 的元素，就把它放到结果前缀末尾。因为 `read` 从左到右扫描，保留下来的元素相对顺序不会变；因为 `write` 永远不超过 `read`，写入不会破坏还没检查的元素。

以 #283 移动零为例，它也是同一个模型，只是最后多了一步：非零元素全部稳定写完后，把剩余位置补成 `0`。

## 复杂度

- 时间复杂度：$O(n)$，每个元素最多被读取一次、写入一次。
- 空间复杂度：$O(1)$，只使用常数个指针和临时变量。

## 易错点

- 把 `write++` 放在赋值前，导致第一个保留元素写错位置。
- 返回了原数组长度，而不是整理后的新长度 `write`。
- #26/#80 去重时只比较 `nums[read]` 和 `nums[read-1]`，没有根据“最多保留几个”来写条件。
- #283 移动零时边扫描边交换，容易破坏非零元素的相对顺序；稳定写入再补零更直接。

## 练习顺序

建议按这个顺序刷：#27, #26, #80, #283, #905, #443。

先用 #27 建立 `read/write` 模型，再做 #26/#80 练去重条件，最后用 #283/#443 处理“前缀写完后还要补后缀或写计数”的变体。
