# 时间版本数据结构：设计与数据结构精讲

## 一句话总结

同一个 key 的历史版本天然按时间递增，查询就是找不超过目标时间的最后版本。

## 识别信号

基于时间的键值存储、快照数组、股票价格波动。

对「时间版本数据结构」来说，识别信号不是关键词本身，而是它们都在要求你把每个操作的时间复杂度要求翻译成内部结构之间的同步关系。读题时先把样例拆成“已经确认”“仍在候选”“答案落点”三块，再决定用哪种结构推进。

## 接口语义锚点

- 本篇围绕「时间版本数据结构」训练：同一个 key 的历史版本天然按时间递增，查询就是找不超过目标时间的最后版本。
- 外部接口看到的是一个结构，内部多个容器必须在每次操作后保持同一份事实。
- 识别信号里的“基于时间的键值存储、快照数组、股票价格波动”要能和代码里的每一次推进对应起来，否则说明你只记住了模板，没有抓住约束。
- 做完 #981、#1146、#2034 时，复盘「接口承诺、内部同步、复杂度来源」是否还能用同一套解释串起来。

## 推导步骤

1. 写入时 append 时间和值。
2. 查询时 upper_bound(timestamp)-1。
3. 快照用版本号。

## 定结构前的边界清单

- 重复插入、删除不存在、容量边界、查询空结构、时间戳相同都要覆盖。
- 对照「时间版本数据结构」标明答案在遍历中何时被写入，避免把中间状态误当最终结论。
- 相关题 #981、#1146、#2034 只保留一个变化点：输入形态、约束或返回值发生了什么改变？
- 若代码需要第二遍、回溯或压缩状态，说明第一遍保留下来的信息刚好够用。

## 易错点

- 时间戳不单调却直接 append。
- 查询早于最早版本没处理。
- 快照 id 和数组下标混淆。
