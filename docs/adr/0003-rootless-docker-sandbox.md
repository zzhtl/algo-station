# ADR 0003：使用 rootless Docker 作为本地判题隔离边界

- 状态：已采纳
- 日期：2026-07-15

## 背景

判题会编译和运行不可信源码。进程级超时无法限制网络、文件系统、进程树和内存，也无法成为足够的隔离边界。项目仍需保持个人电脑可部署，不引入远程沙箱服务。

## 决策

Judge Worker 默认只连接 rootless Docker，并为编译及每个测试用例创建独立容器。每次执行至少包含：

- 禁用网络：`--network none`；
- 根文件系统只读：`--read-only`；
- 丢弃所有 Linux capabilities：`--cap-drop ALL`；
- `no-new-privileges`；
- 容器内使用 UID/GID `65532:65532`；
- 限制 CPU、内存、swap、PID、执行时间和输出大小；
- 只把单次任务临时目录以读写卷挂载到 `/workspace`；
- `/tmp` 使用 `mode=1777` 的限额 tmpfs，容量随内存限制在 64–256 MB 间调整；
- 编译阶段使用 512 MB 内存预算，测试用例运行阶段仍使用题目自己的内存限制；
- 仅用 `--interactive` 转交标准输入，不分配 TTY；
- Docker 参数使用 `Command` 参数数组构造，不把源码或用户输入插入 shell 命令。

Go 和 Rust 使用固定版本的本地 runner 镜像。函数模式由 Worker 生成固定包装器；标准输入输出模式直接编译用户程序。所有练习只允许标准库，不在判题期间访问依赖网络。

## 风险说明

容器不是适合敌对公网多租户的完整安全边界。本实现降低个人自托管环境中的误操作和常见恶意行为风险，但不承诺抵御内核、Docker 或编译器漏洞。

`JUDGE_ALLOW_ROOTFUL=1` 只作为受控环境的显式逃生口，默认关闭。公网部署需要 microVM、独立低权限主机、seccomp/AppArmor、镜像签名、审计与租户级配额等额外措施。

## 被否决方案

- 直接在 Worker 宿主机运行编译器：无法可靠限制文件、网络和进程树。
- 在 HTTP 进程内判题：用户程序故障会影响主站可用性。
- 默认使用 rootful Docker socket：socket 本身近似宿主机 root 权限，不符合最小权限原则。
