# ADR 0002：主站与 Judge Worker 通过 SQLite 租约队列协作

- 状态：已采纳
- 日期：2026-07-15

## 背景

主站需要提供在线判题，但编译和运行用户代码不应发生在 HTTP 进程内。产品定位是个人自托管，默认只有一个主站和一个本机 Worker，不希望引入 Redis、消息队列或额外控制面。

## 决策

使用同一个 SQLite 数据库中的 `submissions` 表作为持久队列：

- HTTP API 只校验练习、语言、契约、64 KiB 源码上限和 20 个活动任务上限，然后写入 `queued`；
- Worker 用单条 `UPDATE ... RETURNING` 原子领取最早任务；
- 领取后写入 `lease_owner`、`lease_until`，并把 `attempts` 加一；
- 执行期间每 10 秒续租，租约默认 45 秒；
- Worker 崩溃后，其他 Worker 可领取过期租约；最多执行 3 次，之后标记 `internal_error`；
- Worker 使用 `judge_workers` 心跳表向界面报告在线状态；
- Accepted 结果写入学习事件，并触发课程完成状态重算。

## 一致性边界

SQLite 是队列与学习状态的唯一持久事实来源。Worker 完成更新只有在任务仍由自己租用时生效，迟到的旧 Worker 不能覆盖新租约结果。

本方案保证至少一次执行，不保证恰好一次。判题执行没有外部副作用，重复执行是可接受的；最终结果更新受租约所有者约束。

## 理由

- 个人部署不需要额外基础设施。
- 数据库重启后任务仍在，主站与 Worker 可以独立升级。
- 租约可以处理 Worker 被杀死、Docker 挂起和主机重启。

## 适用范围与限制

适用于单机、低并发、个人学习。若需要多租户、公网开放或持续高吞吐，应迁移到专用队列、独立数据库、远程执行节点和更强的配额系统，而不是继续扩大 SQLite 方案。
